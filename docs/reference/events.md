# Event Types

OpenParallax uses an event-driven architecture for real-time communication between the Engine and its clients (CLI, web UI, channel adapters). Every significant action in the message pipeline emits an event that clients can react to.

## Overview

Events flow in one direction: **Engine → Client**. The Engine processes messages, evaluates security, executes tools, and streams events back. Clients subscribe to events for a specific session (or globally for system events) and render them appropriately.

There are **7 event types**, each carrying a specific payload. Every event includes a `session_id` for routing — the WebSocket handler filters events so clients only see events for their active session. The exception is `log_entry`, which is global (processed before the session filter).

## Event Types

### `llm_token`

**Emitted when:** The LLM generates a text token during streaming.

**Purpose:** Real-time display of the LLM's response as it's being generated. Clients append each token to the current response buffer.

**Payload:**

| Field | Type | Description |
|-------|------|-------------|
| `token` | string | The text fragment generated by the LLM. Can be a single character, a word, or a partial word — depends on the tokenizer. |
| `session_id` | string | The session this token belongs to. |
| `message_id` | string | The message being responded to. |

**Notes:**
- Tokens arrive rapidly (10-50 per second depending on the provider).
- The full response is assembled by concatenating all tokens. The final text is also sent in the `response_complete` event.
- Tokens may contain partial markdown — clients should handle incremental rendering gracefully.
- Secret redaction happens before emission. If the LLM generates text containing a detected secret (API key, password), the token is redacted before the client sees it.

---

### `action_started`

**Emitted when:** The Engine begins evaluating a tool call proposed by the Agent.

**Purpose:** Tells the client that a tool call is being processed. The client can show a "thinking" or "evaluating" indicator.

**Payload:**

| Field | Type | Description |
|-------|------|-------------|
| `action_id` | string | Unique identifier for this action (ULID-style, time-sortable). |
| `tool_name` | string | The tool being called (e.g., `read_file`, `run_command`, `git_commit`). |
| `arguments` | object | The arguments passed to the tool. May be redacted if they contain secrets. |
| `session_id` | string | Session context. |

**Notes:**
- This event fires *before* Shield evaluation. The action hasn't been approved yet.
- The `action_id` links this event to the subsequent `shield_verdict` and `action_completed` events.

---

### `shield_verdict`

**Emitted when:** The Shield pipeline finishes evaluating a tool call.

**Purpose:** Informs the client of the security decision. Clients can display the verdict (ALLOW/BLOCK with reasoning).

**Payload:**

| Field | Type | Description |
|-------|------|-------------|
| `action_id` | string | Links to the `action_started` event. |
| `decision` | string | `"ALLOW"` or `"BLOCK"`. |
| `tier` | int | Which tier made the final decision (0, 1, or 2). |
| `reason` | string | Human-readable explanation of the decision. |
| `tier_results` | array | Results from each tier that was evaluated. Each entry has `tier`, `decision`, and `reason`. |
| `session_id` | string | Session context. |

**Notes:**
- If the decision is `BLOCK`, no `action_completed` event follows — the action is not executed.
- The `tier` field tells you where the decision was made. Tier 0 decisions are instant (policy match). Tier 2 decisions involve an LLM call and take longer.
- `ESCALATE` is an internal decision — it never reaches the client. An action that escalates from Tier 0 to Tier 1 to Tier 2 will show the final Tier 2 decision.

---

### `action_completed`

**Emitted when:** A tool call has been executed successfully (after Shield approval).

**Purpose:** Delivers the tool execution result to the client. The client can display the output.

**Payload:**

| Field | Type | Description |
|-------|------|-------------|
| `action_id` | string | Links to the `action_started` event. |
| `tool_name` | string | The tool that was executed. |
| `result` | string | The tool's output. For file reads, this is the file content. For commands, this is stdout + stderr. May be truncated for large outputs. |
| `is_error` | bool | Whether the tool execution failed. |
| `duration_ms` | int | How long the execution took in milliseconds. |
| `session_id` | string | Session context. |

**Notes:**
- This event only fires after `shield_verdict` with decision `ALLOW`.
- The `result` is the same content sent back to the Agent for the next LLM round.
- Results may be redacted if they contain detected secrets.

---

### `response_complete`

**Emitted when:** The Agent finishes its response (all tool calls resolved, final text generated).

**Purpose:** Signals that the response is done. Carries the full response text and thought trace.

**Payload:**

| Field | Type | Description |
|-------|------|-------------|
| `content` | string | The complete response text (all `llm_token` values concatenated). |
| `thoughts` | array | Array of `Thought` objects tracing the Agent's reasoning. Each has `stage` ("reasoning" or "tool_call"), `summary`, and optional `detail`. |
| `token_usage` | object | Token counts: `input_tokens`, `output_tokens`, `total_tokens`. |
| `session_id` | string | Session context. |
| `message_id` | string | The message this response answers. |

**Notes:**
- After this event, the client can finalize the response display (stop streaming indicators, enable input).
- The `thoughts` array provides transparency into the Agent's decision process — which tools it called, what it was thinking between calls.
- In normal (non-OTR) sessions, the response is persisted to SQLite and indexed for memory search.

---

### `otr_blocked`

**Emitted when:** A tool call is blocked because the session is in OTR (Off-The-Record) mode.

**Purpose:** Explains to the user why an action was refused in OTR mode.

**Payload:**

| Field | Type | Description |
|-------|------|-------------|
| `tool_name` | string | The tool that was blocked. |
| `reason` | string | Always something like "OTR mode is read-only — [tool] requires write access." |
| `session_id` | string | Session context. |

**Notes:**
- OTR mode filters tools at the definition level — write tools aren't even sent to the LLM. But if the LLM somehow attempts a write action (e.g., through an MCP tool), this event fires.
- The web UI shows OTR blocks with amber styling (matching the OTR accent color).

---

### `error`

**Emitted when:** Something goes wrong in the pipeline.

**Purpose:** Delivers error information to the client for display.

**Payload:**

| Field | Type | Description |
|-------|------|-------------|
| `code` | string | Error code. One of: `LLM_CALL_FAILED`, `STREAM_ERROR`, `CONTEXT_FAILED`, `EXECUTOR_FAILED`, `SHIELD_ERROR`, `INTERNAL`. |
| `message` | string | Human-readable error description. |
| `recoverable` | bool | Whether the session can continue after this error. |
| `session_id` | string | Session context. |

**Error codes:**

| Code | Meaning | Recoverable? |
|------|---------|:------------:|
| `LLM_CALL_FAILED` | Failed to connect to the LLM provider. Network error, invalid API key, rate limit. | Yes |
| `STREAM_ERROR` | LLM stream broke mid-response. Connection dropped, timeout. | Yes |
| `CONTEXT_FAILED` | Failed to assemble the system prompt. Missing workspace files. | No |
| `EXECUTOR_FAILED` | Tool execution threw an unexpected error. | Yes |
| `SHIELD_ERROR` | Shield pipeline itself failed (not a block — an error). Fail-closed: action is blocked. | Yes |
| `INTERNAL` | Unexpected internal error. | Depends |

---

## Event Flow Example

Here's the complete event sequence for a user message that triggers one tool call:

```
User: "Read the file src/main.go"

1. [llm_token]       "I'll read "
2. [llm_token]       "that file "
3. [llm_token]       "for you."
4. [action_started]   tool=read_file, args={path: "src/main.go"}
5. [shield_verdict]   decision=ALLOW, tier=0, reason="read_file allowed by default policy"
6. [action_completed] tool=read_file, result="package main\n\nimport..."
7. [llm_token]       "Here's the "
8. [llm_token]       "contents of "
9. [llm_token]      "`src/main.go`..."
10. [response_complete] content="I'll read that file for you. Here's the contents..."
```

For a blocked action:

```
User: "Delete /etc/passwd"

1. [llm_token]       "I'll try..."
2. [action_started]   tool=write_file, args={path: "/etc/passwd", content: ""}
3. [shield_verdict]   decision=BLOCK, tier=0, reason="write to /etc/* denied by policy"
4. [llm_token]       "That action was blocked..."
5. [response_complete] content="That action was blocked by the security policy..."
```

Note: no `action_completed` event after a BLOCK verdict.

## Transport Mapping

Events are delivered differently depending on the transport:

| Transport | Format | Delivery |
|-----------|--------|----------|
| **gRPC** (CLI) | Protobuf `PipelineEvent` | Server-streaming on `ClientService.SendMessage` |
| **WebSocket** (web) | JSON with `type` and `data` fields | Per-message frames, filtered by `session_id` |
| **Channels** (messaging) | Platform-specific format | Adapted by each channel's `EventSender` |

The `EventSender` interface abstracts this:

```go
type EventSender interface {
    SendEvent(event *PipelineEvent) error
}
```

Every transport implements this single method. The Engine calls `SendEvent` without knowing or caring which transport is receiving the event.

## Subscribing to Events

### EventBroadcaster

The `EventBroadcaster` manages fan-out to multiple clients:

```go
// Subscribe to events for a specific session
ch := broadcaster.Subscribe(sessionID)
defer broadcaster.Unsubscribe(ch)

for event := range ch {
    // handle event
}

// Subscribe to ALL events (global listener)
ch := broadcaster.SubscribeGlobal()
```

- **Session subscriptions** receive events only for their session.
- **Global subscriptions** receive all events (used for log broadcasting).
- Thread-safe with `sync.RWMutex`.
- Non-blocking sends — if a subscriber's channel is full, the event is dropped (prevents slow clients from blocking the pipeline).
